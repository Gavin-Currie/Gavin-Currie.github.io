Git and GitHub


Overview
Git and GitHub is a distributed version control system for managing files and file versions. It's a CLI based program that runs within Terminal (or command prompt), but but you can also use a GUI client version.
(clients include: Tower, GitHub Desktop, Git Kraken among others...)



Terminology:
Some of the jargon, lingo and weird phrases used with Git...

repo: This is short for repository, which is a directory both on your local hard drive and the remote site where your project files and folders are stored.

clone: The activity of copying a project from a remote site (namely GitHub) and downloading it to your local repo folder.The repo you clone, remains seperate and won't affect the repo you copied.

fork: Essentially the same as cloning, but with forking - it's an activity that is unique to Github and only remains on GitHub. There is no fork command you can use on your local directory. However, you can clone that forked repo.

branch: A parallel timeline (or workspace) you can create, where your team can work on the same project files simultaneously without clashing.

origin:The name of your base timeline for your local repo (on your computer)

main: The name of your default base timeline that gets established when you create a repo on GitHub.

How does Git and GitHub relate to each other?
Git is where you produce all the files and folders for your project, and Github is the destination of where to send them.
Think of it this way: Git is your factory, and Github is your warehouse, or distribution centre.

The end goal of Git:
  • To create and develop project files.
  • Refine the code within them and move them to the staging area.
  • Get them from the staging area and/or to the remote repository as finished product.
